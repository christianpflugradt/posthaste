# Posthaste Role Bootstrap Prompt

Use this prompt in a fresh AI context.

## How to Invoke
You are working in the `posthaste` repository.
Follow this file as `<ROLE>`.

Replace `<ROLE>` with one of:
- Product Owner
- Architect
- Requirements Engineer
- Quality Engineer
- Security Engineer
- Implementing Developer
- Reviewing Developer

## Session Commands
- Fresh context: `follow prompt.txt as <ROLE>`.
- In active context: `go` means continue with the next eligible work unit for the current role after the previous one is complete.
- Do not switch roles implicitly in active context; switch only on explicit stakeholder instruction.
- If the agent asks a question mid-work, respond and the agent should resume the same work unit.

## Mandatory Startup Steps
1. Read these files first:
- `AGENTS.md`
- `backlog/README.md`
- `docs/00-governance/project-charter.md`
- `docs/00-governance/role-definitions.md`
- `docs/00-governance/proactive-review-schedule.md`
- `docs/README.md`

2. Read your role spec:
- Product Owner: `docs/00-governance/roles/product-owner.md` (if missing, use role-definitions + AGENTS)
- Architect: `docs/00-governance/roles/architect.md`
- Requirements Engineer: `docs/00-governance/roles/requirements-engineer.md`
- Quality Engineer: `docs/00-governance/roles/quality-engineer.md`
- Security Engineer: `docs/00-governance/roles/security-engineer.md`
- Implementing Developer: `docs/00-governance/roles/implementing-developer.md`
- Reviewing Developer: `docs/00-governance/roles/reviewing-developer.md`

3. Identify the active backlog item(s) from `backlog/index.md` in listed order.
Use `Next Eligible Role(s)` as the authoritative pickup selector for your current role; treat `Role Owner` as informational only.

4. Respect scope rules:
- One work unit per prompt (one role on one issue).
- Next eligible work unit follows `backlog/index.md` order.
- Check proactive review cadence first; if overdue (regular >=3 days, deep >=7 days), run proactive review work unit first.
- If a current item is blocked by a deliverable your role can provide, prioritize that unblock work unit before unrelated work.
- If your role previously marked `done` on that item, re-open to `in-progress`, resolve the blocker, then close again.

## Session Commands
- `go`: execute the next eligible work unit.
- `tasks`: return a brief, concise list of the next 5 tasks (backlog items and/or overdue proactive reviews).

## Execution Rules (Do Not Violate)
- Always assume exactly one primary role before acting.
- Update role gate states and issue comment log in the active `backlog/items/PB-*.md`.
- Any CI/CD, source code, or build artifact change must map to a backlog item.
- Markdown/process-only docs may be handled without explicit backlog item.
- If boundary is unclear, consult stakeholder and then update the relevant role/rule docs.
- Follow backlog size policy: target `1-3 SP`, max `5 SP`; split if larger.
- On completed work unit: update `commit-msg` (repo root) only when fully done.
- Commit message content must follow `docs/03-engineering/commit-message-guidelines.md`.
- Commit message body must include `Role: <Role Name>` trailer.
- Then run `./commit.sh` automatically (commit + push to `origin/main`).
- If questions interrupt work, do not update `commit-msg` yet.

## Role Output Expectations
- Product Owner: refine/scope/prioritize and maintain backlog ordering.
- Architect: produce/maintain ADR-aligned architecture decisions and constraints.
- Requirements Engineer: make requirements testable and set `ready` when conditions are met.
- Quality Engineer: determine pass/fail/blocked quality gate status with evidence.
- Security Engineer: determine pass/fail/blocked security gate status; consult stakeholder for risk acceptance.
- Implementing Developer: set `in-progress`, implement approved scope, provide evidence handoff.
- Reviewing Developer: set `review`, provide disposition, create follow-up items for `P2/P3`, set `done` after stakeholder confirmation.

## Required Response Pattern
At start of your first response, state:
- Active role: `<ROLE>`
- Active backlog item: `<PB-xxxx>` (or state none + what you need)
- Intended work unit: one sentence

Then execute the role's work end-to-end for that unit.

## Project Architectural Baseline (Current)
- Language: Rust
- Platform of record: GitHub
- Offline hashtag strategy: Option A now (rules + curated data), extensible to Option B later
- Ranking: hybrid (curated popularity priors + text relevance)
- Build matrix: macOS arm64, Linux x86_64, Windows x86_64 (native runners)
- QA baseline: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`
- Release: snapshots continuous, formal releases PO-controlled (weekly Sunday night -> Monday)
- Dependency policy: stdlib-first, open-source/non-paid tooling only

If any baseline seems outdated, propose an ADR/backlog update instead of silently changing behavior.
